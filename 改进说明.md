# 项目整体创新改进说明

## 1. 当前项目研究定位

当前项目的核心研究方向是：面向复杂障碍环境下的多机器人路径规划与避障，构建一种由强化学习引导的改进人工蜂群算法。

现有方法可以概括为：

> CNN-PPO 根据当前地图状态动态选择改进人工蜂群算法中的搜索策略组合，IABC 再根据该策略组合在线优化多个机器人下一步的速度和转角。

从代码结构看，项目已经具备以下基础：

- `map.py`：构建多机器人二维路径规划环境，包含圆形障碍物、矩形障碍物、边界检测、机器人间碰撞检测、奖励函数和目标函数。
- `abc_for_map.py`：实现面向地图动作搜索的改进人工蜂群算法，包含多种搜索策略和策略概率调节接口。
- `CNN_PPO.py`：实现 CNN-PPO，用图像状态输入选择策略 profile。
- `network_train.py`：训练 PPO 调节器。
- `network_test.py`、`All_Compare_Test.py`：测试 PPO-IABC 与其他群智能算法。
- `ablation_experiment.py`：标准化消融框架，对比原始 ABC、固定概率 IABC、人工规则 IABC 和 PPO-IABC。

因此，当前项目已经不是单纯的路径规划算法，而是一个“强化学习 + 群智能优化 + 多机器人避障”的混合智能规划框架。

## 2. 当前创新点与不足

### 2.1 已完成的基础创新

当前代码中已经形成的主要创新点是：

> 使用 PPO 作为高层策略调度器，根据当前环境状态动态调整改进人工蜂群算法中不同搜索策略的选择概率。

传统人工蜂群算法通常使用固定的更新公式或人工设置参数，而当前项目让 PPO 学习选择不同搜索策略的概率组合，使 ABC 在不同地图状态下可以表现出不同的搜索倾向。

这一点可以表述为：

> Reinforcement-learning-guided adaptive artificial bee colony algorithm for multi-robot path planning.

或者中文表述为：

> 面向多机器人路径规划的强化学习引导自适应人工蜂群算法。

### 2.2 当前不足

虽然项目已经具备 PPO-IABC 的基础，但从整体创新性角度看，当前方法仍存在以下不足：

1. PPO 只调节全局策略概率，所有机器人共享同一组概率，缺少对不同机器人局部处境的区分。
2. 机器人间避碰主要依赖斥力项、碰撞惩罚和候选动作修复，尚未显式建模机器人之间的冲突关系。
3. PPO 状态输入主要来自整张地图渲染图像，缺少对多机器人交互结构的显式建模。
4. ABC 搜索策略仍然主要是人工设计，强化学习只负责选择策略 profile，没有深入参与搜索规则构造。
5. 整体方法还偏向“算法组合”，需要进一步升级为具有明确机制贡献的框架。

因此，后续改进不应只停留在增加实验数量，而应围绕“更强的机制创新”展开。

## 3. 推荐的三条创新主线完成情况评审

### 3.1 主线一：GNN-PPO + IABC

#### 当前完成情况

当前尚未完成。

代码中 PPO 的状态输入来自 `env.render()` 渲染出的地图图像，并通过 CNN 提取特征。当前没有图神经网络，也没有显式构建机器人、目标点和障碍物之间的图结构。

当前状态：

- 已有 CNN-PPO。
- 已有多机器人环境状态。
- 未构建图节点。
- 未构建图边。
- 未使用 GNN、Graph Attention 或 Transformer 编码多智能体关系。

#### 当前完成度评估

约为 0%-10%。

该方向目前只具备环境数据基础，但方法本身仍是 CNN-PPO，而不是 GNN-PPO。

### 3.2 主线二：Agent-aware IABC 策略调度

#### 当前完成情况

当前部分具备基础，但没有真正完成。

在 `abc_for_map.py` 中，`set_probability()` 设置的是全局概率：

```python
self.p_1
self.p_2
self.p_3
```

这意味着所有机器人共享同一组搜索策略概率。

虽然每个机器人在候选解中都有独立的速度和转角维度，但策略调度仍然是全局统一的。也就是说，不管某个机器人处在障碍附近、目标附近还是机器人拥堵区域，ABC 内部使用的搜索策略分布都是同一套。

#### 当前完成度评估

约为 20%-30%。

已有基础：

- 多机器人动作向量已经按 agent 编码。
- 终止或碰撞的 agent 会被跳过。
- ABC 搜索可以对不同 agent 的动作维度分别更新。

缺失部分：

- 每个机器人独立的策略概率。
- 根据机器人局部状态计算策略概率。
- 局部状态与 ABC 搜索维度之间的明确绑定。
- PPO 或规则模块输出 agent-specific strategy profile。

### 3.3 主线三：Conflict-aware 协同避碰机制

#### 当前完成情况

当前已有避碰基础，但还不是完整的 conflict-aware 协同机制。

项目中已经包含：

- `map.py` 中的机器人间碰撞检测。
- `map.py` 中的机器人间斥力项。
- `experiment_utils.py` 中的候选动作安全修复。
- `step()` 中的碰撞惩罚。

这些机制可以降低碰撞概率，但它们更像是“被动避碰”：

- 发现距离太近后惩罚。
- 候选动作危险时修复。
- 在目标函数中加入斥力项。

真正的 conflict-aware 协同规划应当显式识别冲突机器人对，并根据冲突类型调整搜索行为。

#### 当前完成度评估

约为 35%-45%。

已有基础：

- 碰撞检测。
- 距离斥力。
- 安全动作修复。
- 机器人间最小距离约束。

缺失部分：

- 冲突机器人对识别。
- 冲突严重程度量化。
- 让行、绕行、等待、优先级等协同行为。
- 将冲突结构反馈到 ABC 搜索公式中。
- 将冲突状态反馈给 PPO 策略调度器。

## 4. 后续总体创新升级方向

建议将当前方法从：

> PPO 调节 IABC 搜索策略概率

升级为：

> 图感知、个体自适应、冲突协同的强化学习引导人工蜂群路径规划框架。

推荐方法名称：

> Graph-based Conflict-aware Reinforcement Learning Guided Artificial Bee Colony

中文名称：

> 面向多机器人路径规划的图感知冲突协同强化学习引导人工蜂群算法。

该方法应围绕三个核心机制展开：

1. 图感知状态编码：显式建模机器人、目标和障碍物之间的关系。
2. 个体自适应策略调度：为不同机器人分配不同搜索策略。
3. 冲突协同搜索机制：识别冲突机器人对，并在搜索过程中主动协调避让。

## 5. 改进主线一：Agent-aware IABC 策略调度

### 5.1 改进目标

将当前全局统一的策略概率：

```text
[p1, p2, p3]
```

升级为每个机器人独立的策略概率：

```text
agent_0: [p1_0, p2_0, p3_0]
agent_1: [p1_1, p2_1, p3_1]
...
agent_n: [p1_n, p2_n, p3_n]
```

这样不同机器人可以根据自身状态选择不同搜索倾向。

例如：

- 远离目标的机器人：更多使用全局引导策略。
- 靠近目标的机器人：更多使用局部精细搜索策略。
- 靠近障碍物的机器人：更多使用避障导向策略。
- 与其他机器人距离过近的机器人：更多使用协同避让策略。

### 5.2 创新意义

当前方法中，PPO 只输出一个全局策略 profile，这会忽略多机器人系统中的个体差异。

Agent-aware IABC 的创新点在于：

> 将 ABC 搜索策略调度从环境级别提升到机器人个体级别，使群智能优化过程具备个体自适应能力。

这比单纯调节全局参数更贴近多智能体路径规划问题本身。

### 5.3 技术实现思路

#### 5.3.1 修改 ABC 概率接口

当前 `abc_for_map.py` 中只有：

```python
def set_probability(self, probabilities):
    ...
```

建议新增：

```python
def set_agent_probabilities(self, agent_probabilities):
    self.agent_probabilities = agent_probabilities
```

其中 `agent_probabilities` 可以是：

```python
{
    "agent_0": [0.6, 0.2, 0.2],
    "agent_1": [0.2, 0.6, 0.2],
    "agent_2": [0.2, 0.2, 0.6],
}
```

#### 5.3.2 在 onlooker 阶段按 agent 选择策略

当前 `generate_solution_for_onlooker()` 中会随机选择若干动作维度，然后根据全局概率选择策略。

改进后，应根据维度所属 agent 获取对应策略概率。

对于维度 `d`：

```text
agent_index = d // 2
agent = env.possible_agents[agent_index]
probabilities = self.agent_probabilities[agent]
```

然后基于该 agent 的概率选择搜索策略。

#### 5.3.3 构造机器人局部状态

每个机器人的局部状态可以包括：

- 到目标点距离。
- 到最近障碍物距离。
- 到最近其他机器人的距离。
- 当前朝向与目标方向夹角。
- 是否处于拥堵区域。
- 最近几步是否路径进展缓慢。
- 当前地图编号或局部障碍密度。

这些特征可以由规则模块生成，也可以作为 PPO/GNN 的输入。

### 5.4 最小可落地版本

第一阶段不必直接训练新 PPO，可以先实现规则版 agent-aware profile：

```text
if nearest_robot_distance < safe_distance:
    probabilities = [0.15, 0.65, 0.20]
elif nearest_obstacle_distance < obstacle_threshold:
    probabilities = [0.20, 0.30, 0.50]
elif distance_to_goal > far_threshold:
    probabilities = [0.60, 0.25, 0.15]
else:
    probabilities = [0.25, 0.25, 0.50]
```

这样可以先验证“个体级策略调度”是否有效，再进一步交给 PPO 学习。

### 5.5 代码改造位置

主要涉及：

- `abc_for_map.py`
  - 新增 `agent_probabilities`。
  - 新增 `set_agent_probabilities()`。
  - 修改 `generate_solution_for_onlooker()`。
- `experiment_utils.py`
  - 新增 `agent_manual_probabilities(env)`。
  - 计算每个 agent 的局部状态。
- `ablation_experiment.py`
  - 增加新算法名，例如 `agent_manual_iabc`、`agent_ppo_iabc`。

## 6. 改进主线二：Conflict-aware 协同避碰机制

### 6.1 改进目标

将当前被动避碰机制升级为显式冲突识别与协同避让机制。

当前方法主要是：

```text
候选动作 -> 检测是否碰撞 -> 惩罚或修复
```

改进后应变为：

```text
识别潜在冲突 -> 判断冲突类型 -> 调整搜索策略 -> 生成协同动作
```

### 6.2 创新意义

多机器人路径规划的关键难点不只是单个机器人避开障碍，而是多个机器人在共享空间中互相影响。

Conflict-aware IABC 的创新点在于：

> 将机器人间冲突关系显式引入群智能搜索过程，使搜索不再只关注单个候选解的目标函数值，而是能够根据冲突结构主动调整机器人之间的运动关系。

### 6.3 冲突类型定义

可以先定义以下几类冲突：

#### 6.3.1 距离冲突

两个机器人当前距离低于安全阈值：

```text
distance(agent_i, agent_j) < safe_distance
```

#### 6.3.2 预测冲突

根据候选动作预测下一步位置，如果下一步距离过近，则认为存在预测冲突。

```text
distance(next_pos_i, next_pos_j) < safe_distance
```

#### 6.3.3 对向冲突

两个机器人相向运动，且目标路径可能交叉。

可通过速度方向夹角判断：

```text
cos(theta_i, theta_j) < negative_threshold
```

#### 6.3.4 交叉冲突

两个机器人路径线段在短时间窗口内交叉。

可检测：

```text
segment(pos_i, next_pos_i) intersects segment(pos_j, next_pos_j)
```

### 6.4 协同避让动作模式

识别冲突后，可以引入协同行为模板：

1. 让行：低优先级机器人减速或停止转向。
2. 绕行：一方增大转角，另一方保持目标方向。
3. 分离：两个机器人向相反侧偏转。
4. 优先通过：距离目标更近或路径更短者优先。
5. 局部重规划：冲突区域内临时增大机器人间斥力权重。

这些行为不一定直接作为最终动作，而可以作为 ABC 搜索中的引导项。

### 6.5 将冲突机制加入 ABC 搜索

可以在 `abc_for_map.py` 中加入 conflict-aware term：

```text
new_solution =
    current_solution
    + global_guidance
    + neighbor_guidance
    + random_perturbation
    + conflict_avoidance_guidance
```

其中 `conflict_avoidance_guidance` 根据冲突机器人对生成。

例如，对于冲突机器人 `agent_i` 和 `agent_j`：

- 如果 `agent_i` 优先级较低，则降低其速度维度。
- 如果两者对向接近，则给双方不同方向的转角偏置。
- 如果某个机器人靠近目标，则保持其方向，另一个机器人绕行。

### 6.6 优先级设计

机器人优先级可以由以下因素决定：

- 到目标距离更近者优先。
- 当前路径更通畅者优先。
- 已经等待较久者优先。
- 编队任务中可指定固定优先级。
- 当前速度较大且转向代价高者优先。

一个简单优先级函数：

```text
priority_i =
    w1 * progress_i
    - w2 * distance_to_goal_i
    - w3 * collision_risk_i
    + w4 * waiting_time_i
```

### 6.7 代码改造位置

主要涉及：

- `experiment_utils.py`
  - 新增 `detect_conflicts(env, candidate_solution=None)`。
  - 新增 `compute_agent_priority(env)`。
  - 新增 `conflict_aware_repair(solution, env)`。
- `abc_for_map.py`
  - 在候选解生成过程中加入 conflict-aware guidance。
  - 在 `best_source()` 或 `best_solution()` 前加入冲突风险评估。
- `map.py`
  - 可新增冲突风险计算函数，但建议先放在 `experiment_utils.py`，避免环境类继续膨胀。

### 6.8 最小可落地版本

第一阶段可以先实现预测冲突修复：

1. 根据候选动作预测所有机器人下一步位置。
2. 找出距离过近的机器人对。
3. 对低优先级机器人执行减速转向。
4. 对高优先级机器人保持原动作。

这比当前统一 fallback 更有协同性。

## 7. 改进主线三：GNN-PPO 状态编码

### 7.1 改进目标

将当前 CNN 图像编码升级为图结构编码。

当前 PPO 输入：

```text
800 x 800 x 3 渲染图像
```

推荐升级为：

```text
机器人节点 + 目标节点 + 障碍物节点 + 关系边
```

### 7.2 创新意义

多机器人路径规划天然是关系问题：

- 机器人与机器人之间存在碰撞风险。
- 机器人与目标之间存在任务关系。
- 机器人与障碍物之间存在避障关系。
- 不同机器人数量下，状态维度会变化。

CNN 从图像中隐式学习这些关系，样本效率较低，也较难泛化到不同机器人数量。

GNN-PPO 的创新点在于：

> 使用图结构显式建模多机器人路径规划中的实体关系，使强化学习调度器能够基于交互结构调节群智能搜索过程。

### 7.3 图结构设计

#### 7.3.1 节点类型

可以设计三类节点：

1. 机器人节点。
2. 目标节点。
3. 障碍物节点。

机器人节点特征：

```text
[x, y, orientation, distance_to_goal, nearest_robot_distance, nearest_obstacle_distance, collision_flag, termination_flag]
```

目标节点特征：

```text
[x, y, target_owner_id]
```

障碍物节点特征：

```text
[x, y, radius_or_width, height, obstacle_type]
```

#### 7.3.2 边类型

可以设计以下关系边：

- robot-robot edge：机器人间距离、相对方向、碰撞风险。
- robot-target edge：机器人到目标距离、目标方向。
- robot-obstacle edge：机器人到障碍物距离、安全裕度。
- obstacle-obstacle edge：通常可省略。

#### 7.3.3 图输出

GNN 编码后可以输出：

1. 全局策略 profile：兼容当前 PPO-IABC。
2. 每个 agent 的策略 profile：配合 Agent-aware IABC。
3. 每个冲突 pair 的避让模式：配合 Conflict-aware IABC。

推荐直接输出每个 agent 的策略 profile：

```text
N agents x 3 strategies
```

### 7.4 技术路线

第一阶段可以不直接引入 PyTorch Geometric，而是手写轻量 attention/message passing，降低依赖复杂度。

例如：

```text
agent embeddings = MLP(agent_features)
relation embeddings = attention(agent_i, agent_j, obstacle_k)
profile_i = softmax(MLP(agent_embedding_i + context_i))
```

如果后续需要更标准的 GNN，可以再引入：

- GCN
- GAT
- GraphSAGE
- Transformer encoder

### 7.5 代码改造位置

主要涉及：

- 新增 `state_encoder.py`
  - 构建图结构状态。
  - 输出 agent feature、edge feature。
- 新增或修改 `GNN_PPO.py`
  - 替代 `CNN_PPO.py` 中的 CNN actor/critic。
- 修改 `network_train.py`
  - 使用结构化状态而非 `env.render()`。
- 修改 `ablation_experiment.py`
  - 添加 `gnn_ppo_iabc` 算法入口。

### 7.6 最小可落地版本

先不做完整 GNN，可以做一个 structured-MLP 版本：

```text
每个机器人局部特征 -> MLP -> agent-specific strategy profile
```

这可以作为从 CNN-PPO 到 GNN-PPO 的过渡版本。

## 8. 三条主线的推荐实施顺序

### 第一阶段：Agent-aware IABC

优先级最高。

原因：

- 改动量相对可控。
- 与当前 ABC 结构最贴近。
- 能快速形成新的机制创新。
- 不依赖复杂神经网络重构。

第一阶段目标：

> 将全局策略概率升级为每个机器人独立策略概率。

建议新增方法名：

> Agent-aware IABC

### 第二阶段：Conflict-aware IABC

优先级第二。

原因：

- 当前已有碰撞检测、斥力项和安全修复。
- 可以在现有机制上扩展冲突识别。
- 能增强多机器人协同属性。

第二阶段目标：

> 显式识别机器人间潜在冲突，并在 ABC 搜索中引入协同避让引导项。

建议新增方法名：

> Conflict-aware IABC

### 第三阶段：GNN-PPO + IABC

优先级第三，但长期价值最高。

原因：

- 改动量最大。
- 需要重新设计状态编码和 PPO 网络。
- 但创新性和泛化能力提升最明显。

第三阶段目标：

> 使用图结构状态编码替代 CNN 图像输入，让 PPO 基于多机器人交互关系输出策略调度。

建议新增方法名：

> Graph-based PPO-IABC

## 9. 最终推荐框架

如果三条主线全部完成，最终框架可以命名为：

> GA-CAPPO-IABC

其中：

- `G`：Graph-based，图结构状态建模。
- `A`：Agent-aware，个体自适应策略调度。
- `C`：Conflict-aware，冲突感知协同避碰。
- `PPO-IABC`：PPO 引导的改进人工蜂群算法。

完整中文名称：

> 图感知个体自适应冲突协同 PPO 引导改进人工蜂群多机器人路径规划算法。

更适合论文标题的表达：

> A Graph-based Conflict-aware Reinforcement Learning Guided Artificial Bee Colony Algorithm for Multi-Robot Path Planning

## 10. 具体创新点表述

后续论文或汇报可以将创新点组织为以下三点：

### 创新点一：个体自适应的群智能搜索策略调度

提出一种 agent-aware strategy scheduling mechanism，使每个机器人根据自身局部状态动态选择不同的 ABC 搜索策略，突破现有方法中所有机器人共享统一搜索参数的限制。

### 创新点二：冲突感知的协同搜索机制

提出一种 conflict-aware cooperative search mechanism，显式识别机器人间潜在冲突，并将冲突关系转化为搜索引导项，使群智能优化过程具备主动协同避让能力。

### 创新点三：图结构强化学习调度器

提出一种 graph-based PPO scheduler，将机器人、目标和障碍物建模为图结构，通过关系推理生成策略调度信号，提高不同机器人数量和复杂地图下的泛化能力。

## 13. GPU 迁移可行性与计算瓶颈改进

### 13.1 当前计算瓶颈判断

当前项目中，PPO 网络本身已经具备 GPU 运行条件。`CNN_PPO.py` 中会检测 CUDA：

```python
device = torch.device('cpu')
if torch.cuda.is_available():
    device = torch.device('cuda:0')
```

因此，神经网络前向、反向传播可以迁移到 GPU。

但项目的主要计算瓶颈并不一定在 PPO 网络，而是在以下 CPU 逻辑中：

1. `abc_for_map.py` 中 ABC 种群搜索大量使用 Python 循环和 NumPy。
2. `map.py` 中 `objective()`、`step()`、碰撞检测、障碍物距离计算大多运行在 CPU。
3. `experiment_utils.py` 中候选解修复、起终点采样、距离计算仍是 NumPy 逻辑。
4. `env.render()` 使用图像渲染，涉及 OpenCV、Gym/Pygame 风格流程，通常仍在 CPU。
5. 每一步都需要执行多次候选解评价，导致 `env.objective()` 被频繁调用，CPU 循环开销很高。

所以当前瓶颈可以概括为：

> PPO 可上 GPU，但 IABC 搜索和环境评价仍主要卡在 CPU。

### 13.2 是否可以迁移到 GPU

结论是：可以部分迁移，且值得做；但不建议一开始把整个环境完整迁移到 GPU。

原因是：

- PPO 网络已经比较容易上 GPU。
- ABC 种群评价可以批量化，非常适合 GPU。
- 距离、斥力、碰撞风险等几何计算可以用 PyTorch tensor 并行。
- 但渲染、Gym/PettingZoo 环境接口、Python 字典状态、逐 agent 循环不适合直接迁移。

因此推荐采用分层迁移策略：

```text
保留环境控制流程在 CPU
将高频候选解评价和几何距离计算迁移到 GPU
将 ABC 种群更新向量化
逐步减少 Python for-loop
```

### 13.3 最优先迁移的部分

#### 13.3.1 批量候选解评价

当前 ABC 每次生成一个候选解后，会调用：

```python
self.env.objective(solution)
```

这会导致大量单样本 CPU 调用。

建议改成批量评价：

```python
objectives = env.objective_batch(candidate_solutions)
```

其中：

```text
candidate_solutions shape = [population_size, agent_num * 2]
```

一次性在 GPU 上计算整批候选解的目标函数值。

这是最值得优先做的 GPU 化改造，因为 ABC 的种群搜索天然适合并行。

#### 13.3.2 机器人间距离矩阵

多机器人间距离可以从双层循环改成矩阵计算。

当前逻辑类似：

```text
for agent_i:
    for agent_j:
        compute distance(agent_i, agent_j)
```

GPU 版本可以变成：

```python
diff = positions[:, :, None, :] - positions[:, None, :, :]
dist = torch.linalg.norm(diff, dim=-1)
```

如果有 batch 维度：

```text
positions shape = [batch, agent_num, 2]
dist shape = [batch, agent_num, agent_num]
```

这样可以同时计算所有候选解下所有机器人对的距离。

#### 13.3.3 障碍物距离计算

圆形障碍物距离也适合批量化：

```text
positions shape = [batch, agent_num, 2]
circle_centers shape = [num_obstacles, 2]
distance shape = [batch, agent_num, num_obstacles]
```

矩形障碍物可以用 broadcast 计算机器人与矩形中心的 x/y 距离，并判断是否落入扩展矩形区域。

#### 13.3.4 ABC 种群更新

当前 ABC 中 employed bee、onlooker bee、scout bee 阶段大量使用 Python 循环。

可以先不完全重写 ABC，而是优先将以下部分向量化：

- 初始种群生成。
- 候选解边界裁剪。
- fitness 计算。
- best source 选择。
- random index 采样。

最终目标是让 ABC 内部主要数据结构从 list of `FoodSource` 变成 tensor：

```text
solutions: [population, agent_num * 2]
fitness: [population]
trials: [population]
```

这会显著减少 Python 对象和循环开销。

### 13.4 不建议优先迁移的部分

以下部分不建议第一阶段迁移到 GPU：

1. `render()`：渲染流程通常不是主要可优化对象，且 GPU 化代价高。
2. Gym/PettingZoo 环境接口：状态字典、agent 字典不适合直接 GPU 化。
3. 起点和终点随机采样：频率较低，不是核心瓶颈。
4. 文件写入、日志统计、结果保存：保持 CPU 即可。
5. Pygame/human render：只用于可视化，不应参与高性能训练主循环。

### 13.5 推荐的迁移路线

#### 阶段一：PPO 推理彻底 GPU 化

确保所有 PPO 输入 state、模型、输出都在同一 device 上。

需要检查：

- `state.to(device)`
- `policy.to(device)`
- checkpoint 加载时的 `map_location`
- 避免频繁 CPU/GPU 来回拷贝

当前代码中部分 state 是：

```python
torch.from_numpy(...).float()
```

建议统一改成：

```python
torch.from_numpy(...).float().to(device)
```

#### 阶段二：新增 GPU 版目标函数

在 `map.py` 或新文件中新增：

```python
objective_batch_torch(candidate_solutions, env_state, device)
```

建议不要直接把现有 `objective()` 改掉，而是新增批量版本，降低破坏风险。

该函数输入：

- 候选动作 batch。
- 当前机器人位置。
- 当前机器人朝向。
- 目标点。
- 圆形障碍物。
- 矩形障碍物。
- collision/termination mask。

输出：

- 每个候选解的 objective。
- 可选输出碰撞风险、机器人间最小距离、障碍物最小距离。

#### 阶段三：GPU 版 ABC fitness

在 `abc_for_map.py` 中新增 GPU fitness 路径：

```python
fitness = self.env.objective_batch_torch(self.solutions_tensor)
```

先保留原始 CPU 版本作为 fallback。

建议新增参数：

```python
use_gpu_eval=True
```

如果 CUDA 不可用或 batch 函数未实现，则自动回退 CPU。

#### 阶段四：Tensor 化 ABC 种群

将 `FoodSource` 对象列表逐步替换为 tensor 数据结构。

推荐新建一个类，而不是直接大改旧类：

```python
class TensorABC:
    ...
```

这样可以保留原始 `ABC` 用于对照和回退。

#### 阶段五：与 Agent-aware / Conflict-aware 合并

GPU 化目标函数可以同时服务于后续创新：

- Agent-aware IABC：批量计算每个 agent 的局部风险。
- Conflict-aware IABC：批量计算机器人间距离矩阵和预测冲突。
- GNN-PPO：直接从 tensor 状态构建图特征。

因此 GPU 迁移不仅是工程优化，也可以支撑整体创新升级。

### 13.6 建议新增文件

为了避免 `map.py` 和 `abc_for_map.py` 继续膨胀，建议新增：

```text
gpu_objective.py
```

负责：

- 将环境状态转成 torch tensor。
- 批量预测候选动作后的机器人位置。
- 批量计算目标距离项。
- 批量计算机器人间斥力项。
- 批量计算障碍物斥力项。
- 批量计算碰撞风险。

还可以新增：

```text
tensor_abc.py
```

负责：

- Tensor 化种群。
- GPU 上批量生成候选解。
- GPU 上批量计算 fitness。
- 输出最优 action solution。

### 13.7 GPU 迁移与创新点的关系

GPU 迁移本身不一定是论文核心创新，但它可以成为方法能力的重要支撑。

可以在论文或汇报中表述为：

> To reduce the computational overhead of online swarm search, we implement a batch-parallel candidate evaluation module on GPU, enabling simultaneous evaluation of population-level multi-robot action candidates.

中文表述：

> 为降低在线群智能搜索的计算开销，设计了面向候选动作种群的 GPU 批量并行评价模块，实现多机器人候选动作的并行目标函数计算。

如果后续将其与 Agent-aware 和 Conflict-aware 结合，可以进一步强调：

> GPU 并行评价使得方法能够在每一步同时评估大量个体级策略组合和冲突规避候选动作，从而支撑实时多机器人规划。

### 13.8 风险与注意事项

GPU 迁移需要注意以下问题：

1. 小 batch 不一定更快。如果种群规模很小，CPU 到 GPU 拷贝开销可能抵消收益。
2. 避免每个候选解单独 `.to(device)`，必须整批搬运。
3. 尽量减少 NumPy 和 Torch 之间反复转换。
4. 渲染图像作为 PPO 输入时，图像预处理仍可能在 CPU，需控制频率。
5. 随机数生成需要统一 seed，否则 CPU/GPU 结果可能不完全一致。
6. Tensor 版本目标函数需要和原 CPU `objective()` 做一致性校验。

### 13.9 推荐结论

GPU 迁移是可行的，但推荐优先迁移“批量候选解评价”和“ABC fitness 计算”，而不是完整迁移整个环境。

推荐路线是：

```text
PPO state/model 上 GPU
-> objective_batch_torch
-> GPU fitness evaluation
-> TensorABC
-> 与 Agent-aware / Conflict-aware / GNN-PPO 合并
```

这样既能解决当前实验计算卡在 CPU 的问题，又能为后续整体创新提供工程基础。

## 14. 总结

当前项目已经具备一个较好的基础创新：

> PPO 自适应调节改进人工蜂群算法的搜索策略，用于多机器人路径规划。

但如果希望进一步提升整体创新性，建议不要只增加实验，而是将方法机制升级为：

> 图感知、个体自适应、冲突协同的强化学习引导群智能路径规划框架。

最推荐的落地顺序是：

1. 先做 Agent-aware IABC。
2. 再做 Conflict-aware 协同避碰。
3. 最后做 GNN-PPO 图结构调度器。

这样既不会推翻现有代码，又能逐步把项目从“PPO 调参 ABC”提升为一个更完整、更有研究辨识度的多机器人智能规划框架。

该方法不再使用渲染图像作为状态输入，而是提取每个机器人的结构化局部特征：

- 当前坐标。
- 目标坐标。
- 到目标距离。
- 最近机器人距离。
- 最近障碍物距离。
- 朝向误差。
- termination/collision flag。

MLP 输出每个 agent 的三策略概率：

```text
agent_i -> [p1, p2, p3]
```

训练命令示例：

```bash
python structured_mlp_train.py --agent-nums 5 10 --map-indices 0 1 2 --steps 1000 --batch-scenarios 8 --output-dir structured_mlp_checkpoints
```

更完整的训练可以提高步数：

```bash
python structured_mlp_train.py --agent-nums 5 10 15 --map-indices 0 1 2 --steps 5000 --batch-scenarios 16 --output-dir structured_mlp_checkpoints
```

评估时加载 checkpoint：

```text
--structured-checkpoint path/to/checkpoint.pth
```

若不提供 checkpoint，则使用随机初始化模型，只用于验证流程打通，不应作为正式实验结果。

当前训练方式是规则蒸馏，不是在线 PPO 微调。它的作用是把人工规则版 Agent-aware IABC 转成可学习调度器，为后续 PPO 微调或 GNN-PPO 做中间过渡。

### GPU 迁移状态

已实现第一阶段 GPU 迁移：

- `gpu_objective.py`
  - `objective_batch_torch(candidate_solutions, env, device=None)`
- `map.py`
  - `objective_batch_torch(candidate_solutions, device=None)`
- `abc_for_map.py`
  - `use_gpu_eval=True` 时，`best_solution()` 和 `fitness_for_abc()` 会优先调用 torch 批量 objective。

当前 GPU 化范围：

```text
candidate solution batch
-> torch tensor
-> batched target distance
-> batched robot-robot distance
-> batched obstacle distance
-> objective values
```

尚未完成完整 TensorABC。也就是说，当前版本已经把高频 objective 评价接入 GPU 路径，但 ABC 种群对象本身仍然是 Python list + FoodSource。下一步若要继续优化计算瓶颈，应将种群数据结构改成：

```text
solutions: [population, agent_num * 2]
fitness: [population]
trials: [population]
```
